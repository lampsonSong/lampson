# Debug

严格按阶段执行，不跳步。**本 skill 只负责诊断和方案，不负责修复。修复交给 code-writing skill。**

## 快速路径（已知报错）

适用于有明确报错信息或 traceback 的情况。

### Phase 1 — 复现

拿到明确的报错信息或异常行为。没有复现就不要给方案。

### Phase 2 — 定位

- 优先看 traceback 最后一行（直接原因）
- 向上追溯调用栈找根本原因
- 检查变量类型和值是否符合预期
- 必要时插入 print/logging 辅助排查
- 只做只读探测（ls、cat、which、env），不要盲目全盘搜索

### Phase 3 — 方案

确认根因后，给出修复方案（改哪个文件、改什么逻辑、为什么这么改）。**不执行修复，交给 code-writing skill。**

## 深度诊断（疑难问题）

适用于没有明确报错、性能回归、难以复现的情况。

### Phase 1 — 建立反馈循环

**这是核心技能。** 如果有快速、确定性的 pass/fail 信号，你就能找到原因。

按优先级尝试：

1. **失败的测试** — 写一个能复现 bug 的测试
2. **HTTP 脚本** — curl 对着 dev server 跑
3. **CLI 调用** — 固定输入，diff 输出
4. **最小化复现** — 砍掉无关部分，缩到最小
5. **二分定位** — git bisect 或逐步排除
6. **差分对比** — 旧版本 vs 新版本，同一输入 diff 输出

在这个阶段投入不成比例的精力。不要放弃。

### Phase 2 — 最小化

把复现条件缩到最小：最少的数据、最短的路径、最简单的场景。

### Phase 3 — 假设

基于复现条件，列出可能的原因。按可能性排序。

### Phase 4 — 验证

逐个验证假设。用 instrument（日志、断点、计数器）而非猜测。

### Phase 5 — 方案

确认根因后，给出修复方案（改哪个文件、改什么逻辑、为什么这么改）。**不执行修复，交给 code-writing skill。**

## 交接

诊断完成后，把方案交给 code-writing skill 执行。交接内容：
- 根因描述
- 需要改的文件和具体位置
- 修复逻辑
- 需要的测试验证

## 收尾

- 清理诊断过程中产生的临时文件
- 如果诊断方法有通用性，考虑沉淀到 skill

## 禁止

- 禁止在本 skill 里动手修复代码
- 禁止用临时脚本（/tmp/xxx.py）绕过问题
- 禁止用 workaround、临时 patch 掩盖症状
- 同一任务连续失败 2 次以上，停下来分析根因
- 必须找到根本原因后再给方案
