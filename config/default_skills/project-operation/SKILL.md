## 项目操作前先查记录

对已知项目执行任何操作前，**必须先调 `search_projects` 或 `project_context` 查项目记录**，拿到准确路径和部署信息后再行动。禁止靠关键词猜测路径。

### 步骤
1. 收到项目相关任务 → 先调 `search_projects` 或 `project_context`
2. 从记录中获取路径、部署位置等信息
3. 路径在远程机器 → SSH 过去操作
4. 路径在本地 → 直接操作
5. 操作后校验：看一眼 git remote / README / 目录结构，确认是正确的项目

### 流程反思（2026-04-29 犯错复盘）
用户说"拉模型平台代码"，我犯了三个错：
1. **跳过查记录** — 没调 search_projects，靠关键词猜了一个本地路径 `bm_ai_data_scope`
2. **没有验证** — `bm_ai_data_scope` 跟"模型平台"完全不搭，但我没质疑
3. **过度自信** — 拉完还一本正经列改动，完全没有意识到搞错了项目

根因：把"搜索"当成可选步骤，觉得"大概知道在哪"就跳过了。但实际上项目路径可能在本机、可能在远程、可能有多个同名目录，不查记录就是猜。

## 更新项目知识时必须二次确认项目身份

当用户提到某个项目名称（如"这是 lampson 的项目"），在写入或更新任何项目知识文件之前：
1. 先明确用户说的是哪个项目
2. 确认要操作的目标项目与用户指定的一致
3. 禁止凭上下文惯性假设目标项目
