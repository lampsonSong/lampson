# Diagnose

严格按阶段执行。跳过阶段必须有明确理由。

参考自 Matt Pocock 的 diagnose skill，针对 agent 工作场景调整。

## Phase 1 — 建立反馈循环

**这是核心技能。** 如果有快速、确定性的 pass/fail 信号，你就能找到原因。没有的话，看再多代码也没用。

按优先级尝试：

1. **失败的测试** — 在合适的层级写一个能复现 bug 的测试
2. **HTTP 脚本** — curl 对着 dev server 跑
3. **CLI 调用** — 固定输入，diff 输出
4. **最小化复现** — 砍掉无关部分，缩到最小
5. **二分定位** — git bisect 或逐步排除
6. **差分对比** — 旧版本 vs 新版本，同一输入 diff 输出

在这个阶段投入不成比例的精力。不要放弃。

## Phase 2 — 最小化

把复现条件缩到最小：最少的数据、最短的路径、最简单的场景。

## Phase 3 — 假设

基于复现条件，列出可能的原因。按可能性排序。

## Phase 4 — 验证

逐个验证假设。用 instrument（日志、断点、计数器）而非猜测。

## Phase 5 — 修复

确认根因后修复。修复后：

1. 确认原 bug 复现步骤不再触发
2. 写回归测试防止再次出现
3. 清理调试产物

## 规则

- 同一任务连续失败 2 次以上，停下来分析根因
- 先做只读探测收集信息，不要上来就改代码
- 所有方案都失败时，明确告诉用户卡在哪里
