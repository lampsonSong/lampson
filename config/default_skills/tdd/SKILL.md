# Test-Driven Development

## 核心原则

测试验证行为，不验证实现。好的测试像规格说明——"用户可以用有效购物车结算"。改代码不破坏测试；改测试只因为行为变了。

## 红绿重构循环

**纵向切片，不要横向平铺。** 一次只做：

1. **RED** — 写一个失败的测试，描述一个行为
2. **GREEN** — 写最少的代码让测试通过
3. **REFACTOR** — 在测试保护下重构

然后重复。每个测试响应上一个循环中学到的东西。

## 反模式：横向切片

禁止先写完所有测试再写实现。这会产生垃圾测试：

- 批量写的测试验证的是想象中的行为，不是实际行为
- 测试变成对"形状"的测试（数据结构、函数签名），而非用户行为
- 重构时测试该挂不挂、不该挂的挂了

## 好测试的特征

- 通过公开接口测试，不测私有方法
- 不 mock 内部协作者（mock 外部依赖可以）
- 读起来像规格说明
- 重构实现时不影响测试

## 运行

```bash
# Python
pytest -xvs tests/test_xxx.py

# 带覆盖率
pytest --cov=src/module tests/
```
