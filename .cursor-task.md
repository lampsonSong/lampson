# Task: 实现 Context Compaction（记忆归档压缩）

## 背景

阅读 `docs/compaction-design.md`，这是 Lampson 的上下文压缩设计方案。

核心思想：**归档优于丢弃**。当对话 token 超过阈值时，不直接截断，而是：
1. 把有长期价值的内容归档到 skill/project 文件
2. 对剩余内容生成结构化摘要
3. 用摘要重置对话窗口

## 你需要做的

### 第一步：Review 设计文档

读 `docs/compaction-design.md`，结合现有代码结构，思考以下问题：

1. **接入点**：压缩应该在 Agent.run() 的哪里触发？设计文档说"LLM 回复结束后检查"，但当前 Agent.run() 没有 token 计数能力。怎么获取当前 context token 数？
   - 选项A：LLM 返回的 `usage` 字段里有 `total_tokens`
   - 选项B：自己估算（tiktoken 或粗略计算）
   - 建议：先用 usage 字段，最简单

2. **context window 大小**：GLM-5.1 的 context window 是多少？设计文档的 `triggerThresholdPercent` 需要这个值。建议做成配置项。

3. **LLM 调用次数**：归档阶段需要多次 LLM 调用（分类、判断归档目标、整合 skill/project 内容、生成摘要）。这对 MVP 来说可能偏重。建议：
   - MVP 先做**单阶段摘要**：把整个对话历史交给 LLM，一次生成"归档内容 + 结构化摘要"
   - 后续迭代再拆分多阶段

4. **Skill/Project 写入**：设计文档要求"合并/更新/淘汰"已有内容，这需要 LLM 读取已有文件 + 新内容 → 重新整合。MVP 可以简化为追加（append），后续再做智能合并。

5. **stopReason**：设计文档要求只有 `end_turn` / `aborted` 才触发。但 OpenAI SDK 的 finish_reason 值是 `stop` / `tool_calls`，不是 `end_turn`。需要映射。
